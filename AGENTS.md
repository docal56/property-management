# AGENTS.md

Entry point for coding agents working in this repo. Read this first, every time.

---

## 1. What this repo is

The codebase for an AI phone agent product built for UK estate agents. Inbound tenant maintenance calls are answered by an AI voice agent (ElevenLabs), and the resulting calls and Issues are surfaced in a simple web app that the Agency's Admin uses to triage and resolve them. The app is multi-tenant — each Agency sees only their own data.

- GitHub: https://github.com/docal56/property-management (public)
- Full product context: `docs/phase-0/product-spec.md`

The current active phase is **Phase 0**. Everything in this repo should be scoped to Phase 0 unless a spec says otherwise.

## 3. Stack summary

Every tool is installed at **latest stable at time of install** per `DECISIONS.md` → "Stack versioning policy". Do not downgrade majors without adding a DECISIONS entry explaining why.

- **Frontend:** Next.js (App Router), React, TypeScript strict
- **Styling:** Tailwind v4 with CSS-first `@theme` config in `app/globals.css` (no `tailwind.config.js`); Radix UI primitives for accessible components
- **Backend:** Convex (database, server functions, file storage, scheduled jobs, HTTP actions)
- **Auth:** Clerk (`@clerk/nextjs`), custom tenancy via `publicMetadata.agencyId`
- **Hosting:** Vercel (Next.js) + Convex Cloud (backend)
- **AI Phone Agent:** ElevenLabs (managed outside this app; posts webhooks in)
- **Telephony:** Twilio numbers wired to ElevenLabs (this app does not talk to Twilio directly in Phase 0)
- **Package manager:** pnpm. Commit `pnpm-lock.yaml`. Delete any other lockfiles if they appear.
- **No `src/` directory.**

## 5. Critical rules (non-negotiable)

1. **Every tenant-scoped Convex query/mutation calls `getAgencyIdForCurrentUser(ctx)` first and filters by that `agencyId`.** No exceptions. If you skip this you leak data across Agencies.
2. **No Users table.** Clerk is the source of truth for users. Only Clerk user IDs (strings, e.g. `user_xxx`) land on records.
3. **No Next.js API routes.** Webhooks and any external-facing HTTP goes through Convex HTTP actions (`convex/http.ts`).
4. **No onboarding UI. No Team Member role. No filters on Open Issues. No Managed Properties, Contacts, or Settings pages.** All deferred post-Phase 0.
5. **`urgent` is AI-set on creation.** No UI for the Admin to set it. The Team Member can override the flag but not via a dedicated "set urgency" control.
6. **"Resolved" is the only end state.** No separate "Closed-No-Action" status.
7. **TypeScript strict mode on.** No `any` without a comment justifying why.
8. **Component files stay under ~200 lines.** Split if larger.
9. **No `console.log` in committed code.** `console.log` inside a Convex function is fine (it goes to Convex logs); remove everywhere else before commit.
10. **Tailwind v4 only.** Tokens live in `@theme` in `app/globals.css`. No `tailwind.config.js` should ever exist in this repo.
11. **Repo is public.** Never commit secrets, `.env.local`, real Agency data, or real Clerk/Convex/ElevenLabs keys.

## 6. Testing

- **TDD for Convex functions.** Use `convex-test`. Write the test, then the function. Every PR includes tests for the Convex code it introduces.
- **Vitest** for any non-Convex logic.
- **No Playwright / E2E** in Phase 0.
- **UI is manually verified.** Coding agents are trusted to implement UI without tests in Phase 0.

What gets tested:

- Queries: correct data returned, correct scoping by Agency, correct filtering
- Mutations: correct state change, correct authorization
- Webhook action: valid payload creates Issue; invalid returns 400; missing secret returns 401; failed calls (missing fields) create a Call without an Issue

Critical flow integration tests:

- ElevenLabs webhook → Call created → Issue created → appears in `issues.listOpen`
- Media upload → attached to Issue → appears in Issue detail query
- Media retention job → marks → deletes

## 7. Before calling work done

Self-review is the last step before reporting a task complete or asking for a merge. Do not skip it, and do not rely on your own confidence that the code is fine.

Work through `lessons.md` → "Standing review checklist" in full. In particular:

1. **Look at the PR, not just the code.** `gh pr checks <N>` first. A failing required check is a blocker — investigate before reviewing the diff. "Builds locally" is not evidence that CI will pass.
2. **Trace every `process.env.*` read.** Does the variable exist in every environment this code executes in (local dev, CI, preview deploy, production)? If not, describe the failure mode before calling the code done.
3. **Prove each acceptance criterion.** For every bullet in the ticket, point at the file, test, or command output that demonstrates it's met.
4. **Scan for secrets** before pushing — never commit real keys, real tenant data, or anything that would leak if the repo is public.

When a review misses something — a failing check, a broken preview, an edge case that bit you after "done" — append an entry to `lessons.md` with the check that would have caught it. The list only works if it grows.

## 8. How to append to DECISIONS.md

If during implementation you make a non-obvious call — downgrading a version, picking one pattern over another, working around a platform limitation — append an entry to `DECISIONS.md`. Future agents and reviewers need to know why.

Format (from `DECISIONS.md` header):

```
## YYYY-MM-DD — [short title]
**Context:** what prompted this
**Decision:** what we're doing
**Why:** reasoning
**Watch for:** gotchas or implications
```

Entries are chronological, newest at the bottom of the "Entries" section.
