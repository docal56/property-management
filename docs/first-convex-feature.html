Walkthrough: your first real feature on Convex + Clerk

Using "Issues" (the existing mock UI at (app)/page.tsx) as a concrete example. Every pattern below repeats for future features.

Wiring this feature touches everything you'll use repeatedly — schema, org scoping, reactive queries, mutations.

Shape of a feature on this stack

Four files, typically:

  1. convex/schema.ts — add the table.
  2. convex/issues.ts — queries + mutations.
  3. convex/lib/auth.ts — shared "resolve current user + org" helper (used by every feature).
  4. src/app/(app)/page.tsx — wire the existing UI to useQuery / useMutation.

1. Schema

// convex/schema.ts (extend the existing)
issues: defineTable({
  orgId: v.id("orgs"),
  createdBy: v.id("users"),
  assignedTo: v.union(v.id("users"), v.null()),
  title: v.string(),
  description: v.union(v.string(), v.null()),
  status: v.union(
    v.literal("open"),
    v.literal("resolved"),
    v.literal("archived"),
  ),
})
  .index("by_org_and_status", ["orgId", "status"])
  .index("by_assignee", ["assignedTo"]),

What matters here vs. MySQL thinking

  • FK types are v.id("orgs") / v.id("users") — typed, not just strings. _id is opaque and globally unique across Convex.
  • Index for every query shape. No "the planner will figure it out." The indexes listed are the ones you can query with. Keep them minimal — add only what a handler actually needs.
  • No nulls where defaults work; use v.union(..., v.null()) when nullable.

2. The auth helper (reused by every feature)

// convex/lib/auth.ts
import type { QueryCtx } from "../_generated/server";

export async function requireUserAndOrg(ctx: QueryCtx) {
  const identity = await ctx.auth.getUserIdentity();
  if (!identity) throw new Error("Not authenticated");

  const user = await ctx.db
    .query("users")
    .withIndex("by_clerk_id", (q) => q.eq("clerkId", identity.subject))
    .unique();
  if (!user) throw new Error("User not synced yet");

  // Clerk v2 session token puts the active org at `identity.o`
  const orgCtx = (identity as unknown as { o?: { id: string } }).o;
  if (!orgCtx) throw new Error("No active organization");

  const org = await ctx.db
    .query("orgs")
    .withIndex("by_clerk_id", (q) => q.eq("clerkId", orgCtx.id))
    .unique();
  if (!org) throw new Error("Org not synced yet");

  return { user, org };
}

Why this pattern

  • Every authenticated feature function starts with const { user, org } = await requireUserAndOrg(ctx).
  • Throws bubble up and Convex returns an auth error to the client — no partial writes, no silent data leaks across orgs.
  • Reads the Clerk v2 compact org context (identity.o) — same one seen in the JWT. No extra DB call needed to know which org is active.

3. Queries + mutations

// convex/issues.ts
import { v } from "convex/values";
import { mutation, query } from "./_generated/server";
import { requireUserAndOrg } from "./lib/auth";

export const listOpen = query({
  args: {},
  handler: async (ctx) => {
    const { org } = await requireUserAndOrg(ctx);
    return await ctx.db
      .query("issues")
      .withIndex("by_org_and_status", (q) =>
        q.eq("orgId", org._id).eq("status", "open"),
      )
      .order("desc")
      .collect();
  },
});

export const create = mutation({
  args: {
    title: v.string(),
    description: v.optional(v.string()),
  },
  handler: async (ctx, { title, description }) => {
    const { user, org } = await requireUserAndOrg(ctx);
    return await ctx.db.insert("issues", {
      orgId: org._id,
      createdBy: user._id,
      assignedTo: null,
      title,
      description: description ?? null,
      status: "open",
    });
  },
});

export const updateStatus = mutation({
  args: {
    id: v.id("issues"),
    status: v.union(
      v.literal("open"),
      v.literal("resolved"),
      v.literal("archived"),
    ),
  },
  handler: async (ctx, { id, status }) => {
    const { org } = await requireUserAndOrg(ctx);
    const issue = await ctx.db.get(id);
    if (!issue || issue.orgId !== org._id) throw new Error("Not found");
    await ctx.db.patch(id, { status });
  },
});

Things worth noting

  • Authorization is scoped by org at query time, not by app-layer checks. The .withIndex("by_org_and_status", q => q.eq("orgId", org._id)...) means a user can only ever see their org's rows — enforced at the DB layer.
  • updateStatus double-checks issue.orgId === org._id before patching. Always verify ownership on mutations that take an _id — the client could pass any id.
  • Transactional by default — the whole handler is one transaction. No race conditions between the read and the patch.

4. React wiring

// src/app/(app)/page.tsx (sketch — replacing the mock data)
"use client";
import { useMutation, useQuery } from "convex/react";
import { api } from "../../../convex/_generated/api";

export default function OpenIssuesPage() {
  const issues = useQuery(api.issues.listOpen);
  const createIssue = useMutation(api.issues.create);
  const updateStatus = useMutation(api.issues.updateStatus);

  if (issues === undefined) return <LoadingSkeleton />;

  return (
    <div>
      <button
        type="button"
        onClick={() =>
          createIssue({ title: "New issue", description: "From the UI" })
        }
      >
        New issue
      </button>
      <ul>
        {issues.map((issue) => (
          <IssueListRow
            key={issue._id}
            issue={issue}
            onResolve={() =>
              updateStatus({ id: issue._id, status: "resolved" })
            }
          />
        ))}
      </ul>
    </div>
  );
}

What's happening

  • useQuery(api.issues.listOpen) subscribes. It's not a one-shot fetch — Convex pushes updates over the WebSocket whenever any data the query reads changes. You never refetch(). If two users are looking at the list and one resolves an issue, the other's UI updates within a tick.
  • issues === undefined means "loading." issues === [] means "loaded, empty."
  • Mutations return a promise. You can await them for error handling; UI updates from the pending query subscription, not the mutation's return value.

What this gives you that MySQL didn't

  • No cache invalidation. Ever. Convex tracks which rows each query reads; if any of those change, the query re-runs for every subscribed client.
  • No optimistic UI gymnastics. Most of the time, the subscription update (~50–150ms round-trip) is fast enough that you don't notice. For very snappy UX, Convex has optimistic updates as a first-class feature, but skip it until you miss it.
  • Org tenancy is a query constraint, not a middleware layer. The withIndex clause IS the authorization boundary. No "accidentally return other org's rows" class of bugs if you discipline yourself to always filter by org._id.

Order to build in

  1. Add issues to schema, write listOpen + create. Replace the page's mock data. Ship.
  2. Use the live feature, see what's awkward. Usually: search, filters, pagination, detail view.
  3. Add getById query (similar org-scoped auth check).
  4. Add status filters / assignments as new mutations.
  5. Maybe: attach to a call via issue.callId: Id<"calls"> when you build calls.
Start super thin. listOpen + create + the UI wired up is maybe an hour. That gives you the complete pattern for every feature after.
