# Design Spec

Feature and page breakdown for **Phase 0** of the app. Each page describes its purpose, sections, UX details, and open questions for design exploration.

Phase 0 scope is defined in `product-spec.md`. This spec covers the two pages in Phase 0:

1. Open Issues (with Issue slide-over panel)
2. Call Logs

Later phases will add: Managed Properties, Contacts, Settings, empty/loading/error state designs, Team Member features.

This spec is **direction, not pixel specification.** Visual design lives in Figma. Open questions are listed per page so the implementing agent knows where latitude exists and where decisions are fixed.

---

## Shell (context only — already designed in Figma)

Phase 0 has a collapsible left sidebar shell with:
- Agency branding top-left
- Two nav destinations: Open Issues, Call Logs
- Global search top-right (scope per page, see each page)
- No settings icon in Phase 0

Shell is implemented from Figma. No open questions here.

## 1. Open Issues

### Purpose

The primary workspace for the Admin. Shows every open Issue grouped by status so the Admin can triage, act, and close the loop.

### Sections

**Page header**
- Page title: "Open Issues"
- Global search input, top-right. Searches across property address and issue description within open issues.

**Status-grouped list**

Issues are grouped into collapsible sections by status, in this order:
- New Issue
- In Progress
- Contractor Scheduled
- Awaiting Follow up
- Closed (collapsed by default — contains all Issues with status = Resolved)

Each group is an expand/collapse panel. Groups with zero Issues still render (empty groups are shown as collapsed, empty). Groups expand by default except "Closed".

**Row content (per Issue)**
- Checkbox (visible on row hover or selection)
- Property address
- Truncated issue description (single line, ellipsis at container edge)
- Urgent badge (red pill, label "Urgent") — shown only when the Issue is flagged urgent
- Timestamp (time-of-day for today; date otherwise — TBD in implementation)
- Arrow indicator on hover, signalling the row opens the panel

**Row interactions**
- Click row → opens the Issue slide-over panel (see section 2)
- Keyboard: while the panel is open, up/down arrows move between Issues

### UX details

- Grouping communicates status; no separate status column or inline label on the row.
- No filters in Phase 0 — search only.
- Search is fast and local to currently loaded Issues (no debounce needed at MVP scale).
- Closed section is collapsed by default to keep the active work front-and-centre; expanding it reveals Resolved Issues.
- The row container should tolerate long descriptions gracefully — always truncate, never wrap.

### Open questions

- Timestamp format on rows (time for today, date otherwise, or always relative like "2h ago"?) — decide in implementation
- Behaviour of the checkbox column in Phase 0. The design shows it but no bulk actions exist. Options: (a) remove the checkbox for Phase 0 since it does nothing, (b) keep it for visual consistency with later phases. Defer to implementation.
- Empty group visual treatment — thin collapsed bar, hidden entirely, or always shown? Not critical for Phase 0.

## 2. Issue slide-over panel

### Purpose

The workspace for a single Issue. Gives the Admin everything they need to understand the issue, take action, and update state — without losing their place in the Open Issues list.

### Panel behaviour

- Slides over from the right
- The Open Issues list remains visible to the left (partially, depending on panel width)
- Up/down arrow buttons in the panel header move to the previous/next Issue in the currently filtered Open Issues list
- Keyboard: up/down arrow keys mirror the arrow buttons when the panel is focused
- Close button (X) dismisses the panel and returns focus to the triggering row
- Opening the panel updates the URL so Issues are linkable/shareable (exact URL pattern TBD in tech spec)

### Sections

**Panel header**
- Property address (primary title)
- Up arrow, down arrow (browse), close button — top-right

**Action fields**
- Status dropdown (options: New Issue, In Progress, Contractor Scheduled, Awaiting Follow up, Resolved)
- "Contractor for the job" free-text input — Admin types the contractor's name manually. Not a picker, not a reference to another entity.

**Details block**
- Heading: "Details"
- Structured text block generated by the AI Agent containing:
  - Address
  - Name (tenant)
  - Contact Number (tenant)
  - Issue description
- This block is the canonical copyable context for the Issue
- Buttons below the Details block:
  - Upload Media (left, secondary) — opens file picker for photos/videos
  - Copy Details (right, primary) — copies the entire Details block to the clipboard

**Call recording**
- Heading: "Call recording"
- Audio player with play/pause, scrub bar, duration
- Plays the ElevenLabs call recording

**Transcript**
- Heading: "Transcript"
- Full conversation rendered as chat bubbles
- AI Agent messages styled differently from tenant messages (e.g. left-aligned greyed vs. right-aligned blue)
- No truncation — the panel scrolls as needed

### UX details

- The Details block must read naturally as a message — when the Admin clicks "Copy Details", the clipboard content is ready to paste into WhatsApp, SMS, or email to a contractor. This shapes how the AI Agent prompt must write this block.
- Upload Media accepts photos and videos. File size and type limits TBD in tech spec.
- Uploaded media appears inline (thumbnail grid or similar) near the Upload Media button after successful upload. Click thumbnail to enlarge. Exact layout deferred to implementation.
- The panel is tall and scrollable. Action fields (Status, Contractor) stay at the top so they're always reachable when the Admin lands on a new Issue via up/down browsing.

### Open questions

- Where uploaded media thumbnails render inside the panel — implementation choice
- Whether the Upload Media control shows upload progress or just an immediate thumbnail once done
- Transcript styling for AI Agent vs. tenant bubbles — follow the design system when it exists
- Panel width — specified in Figma; locked there
- Whether closing the panel clears keyboard focus or returns it to the row that opened it (default: return to row)

## 3. Call Logs

### Purpose

Reporting view for the Admin. Shows how the Phone Agent is performing and how much call volume the Agency has over a chosen window.

### Sections

**Page header**
- Page title: "Overview"
- Global search input, top-right (searches call logs — exact scope TBD)
- Date range selector, top-right (default: "This week")

**Metrics row**

Four metric cards across the top:
- **Calls** — total calls in the date range
- **Issues Recorded** — calls that produced an Issue (i.e. at least reached the system)
- **Open Issues** — number of Issues currently open, regardless of date range (this metric is a live count, not tied to the range)
- **Pass Rate** — Success calls ÷ Total calls, in the date range, expressed as a percentage

**Logs table**
- Heading: "Logs"
- Columns: Date, Agent, Duration, Call Status
- Each row represents one call
- Call Status cell: "Success" (green pill) or "Failed" (red pill)
- Table respects the selected date range

### UX details

- The date range selector updates metrics and the logs table together. No partial application.
- "Agent" column is the name of the Phone Agent that handled the call. In Phase 0 there is only one (Property Management) so this column shows the same value for every row — worth retaining for future proofing.
- Rows are not clickable in Phase 0. Revisiting a specific call's issue record is a post-Phase-0 feature.
- "Open Issues" metric is not time-bound. The other three are.

### Open questions

- Whether to include a "Failed" hover/tooltip explaining why the call failed (which of the four fields was missing). Nice-to-have; defer.
- What the default date range should be if the Admin hasn't set one. Design shows "This week" — keep as default.
- Pagination or virtualisation for very long logs tables — not needed at MVP scale; revisit when needed.

## Cross-cutting notes

**Responsive behaviour:** Phase 0 targets desktop and tablet primarily; mobile web is a nice-to-have but not a blocker. The slide-over panel on mobile would become full-screen; the table on Call Logs would need a condensed format. Defer mobile details to implementation unless the first Agency explicitly needs mobile day one.

**Urgent display:** the Urgent flag (boolean on Issue) is shown as a red "Urgent" pill. Consistent wording across list rows and panel.

**Data that exists but has no Phase 0 UI:**
- Managed Property records (created during onboarding, referenced by Issues)
- Contact records (created during onboarding, referenced by Issues)
- Tenant history, property history, issue history across entities
- Audit trail of changes to an Issue

These exist in the data model per the product spec but have no dedicated UI in Phase 0.
