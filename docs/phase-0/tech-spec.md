# Tech Spec — Phase 0

How Phase 0 is built. This spec is the bridge between `product-spec.md` (what) and `design-spec.md` (how it looks) and implementation.

Coding agents should load this in full when scaffolding the project, and selectively by section for specific work.

---

## 1. Stack

**Versioning policy:** every tool below is pinned to "latest stable at time of install." Do NOT downgrade to an older major unless a specific incompatibility is documented in DECISIONS.md. See DECISIONS.md — "Stack versioning policy" for the rationale.

- **Frontend framework:** Next.js (App Router), React, TypeScript — all latest stable
- **Styling:** Tailwind CSS (latest stable — v4+) with its CSS-first `@theme` config; Radix UI primitives for accessible components
- **Backend:** Convex (database, server functions, file storage, scheduled jobs, HTTP actions) — latest stable
- **Auth:** Clerk (custom tenancy model — see section 5) — latest stable `@clerk/nextjs`
- **Hosting:** Vercel for the Next.js app; Convex Cloud for the backend
- **AI Phone Agent:** ElevenLabs (managed outside this app; posts webhooks into it)
- **Telephony:** Twilio numbers are wired to ElevenLabs — this app does not integrate directly with Twilio in Phase 0

Package manager: pnpm (latest stable). Lockfile committed.

## 2. Project Setup

New repo, starting from zero. First coding agent scaffolds the project.

### 2.1 Initialisation sequence

1. `npx create-next-app@latest` — TypeScript, App Router, Tailwind, no `src/` directory, import alias `@/*`. Uses the latest stable Next.js / React / Tailwind at time of install.
2. `npx convex@latest dev` — initialise Convex, create dev deployment, install Convex agent (say yes when prompted)
3. Install the Vercel plugin for coding agents (see `docs.md` → Vercel)
4. Install Clerk: `pnpm add @clerk/nextjs` and configure
5. Install Clerk Auth Components skill for coding agents (see `docs.md` → Clerk)
6. Create folder structure per section 2.2
7. First commit: working Next.js app + Convex connected + Clerk auth scaffold
8. Deploy to Vercel (main branch) + Convex Cloud (production deployment)

**Tailwind v4 note.** From Tailwind v4 onwards, configuration is CSS-first via `@theme` in `app/globals.css` — there is no `tailwind.config.js`. Design tokens go inside `@theme { --color-*: ...; --font-*: ...; }` and are automatically available as Tailwind utility classes. See `docs/design-system-reference.md` for the exact tokens and setup.

### 2.2 Folder structure (no `src/`)

## 4. Convex Schema

Full schema for Phase 0. Note: field names are camelCase. IDs use Convex's `Id<"tableName">` type.

```ts
// convex/schema.ts
import { defineSchema, defineTable } from "convex/server";
import { v } from "convex/values";

export default defineSchema({
  agencies: defineTable({
    name: v.string(),
    twilioPhoneNumber: v.string(),
    elevenLabsAgentId: v.string(), // reference to the Property Management agent in ElevenLabs
  })
    .index("by_twilioPhoneNumber", ["twilioPhoneNumber"])
    .index("by_elevenLabsAgentId", ["elevenLabsAgentId"]),

  properties: defineTable({
    agencyId: v.id("agencies"),
    address: v.string(),
    postcode: v.string(),
    currentTenantId: v.optional(v.id("contacts")),
  })
    .index("by_agency", ["agencyId"])
    .index("by_agency_postcode", ["agencyId", "postcode"]),

  contacts: defineTable({
    agencyId: v.id("agencies"),
    name: v.string(),
    phoneNumber: v.string(),
    email: v.optional(v.string()),
  })
    .index("by_agency", ["agencyId"])
    .index("by_agency_phone", ["agencyId", "phoneNumber"]),

  calls: defineTable({
    agencyId: v.id("agencies"),
    elevenLabsCallId: v.string(), // the ElevenLabs identifier for this call
    agentName: v.string(), // in Phase 0 always "Property Management"
    durationSeconds: v.number(),
    callStatus: v.union(v.literal("success"), v.literal("failed")),
    audioUrl: v.optional(v.string()), // ElevenLabs-hosted URL, or Convex storage ID once uploaded
    transcript: v.array(
      v.object({
        speaker: v.union(v.literal("agent"), v.literal("caller")),
        text: v.string(),
      })
    ),
    issueId: v.optional(v.id("issues")), // set only when callStatus = success
    startedAt: v.number(), // unix ms — when the call happened
  })
    .index("by_agency", ["agencyId"])
    .index("by_agency_startedAt", ["agencyId", "startedAt"])
    .index("by_elevenLabsCallId", ["elevenLabsCallId"]),

  issues: defineTable({
    agencyId: v.id("agencies"),
    propertyId: v.id("properties"),
    contactId: v.id("contacts"), // the tenant who reported the issue
    callId: v.id("calls"),
    details: v.object({
      address: v.string(),
      tenantName: v.string(),
      tenantContactNumber: v.string(),
      description: v.string(),
    }),
    urgent: v.boolean(),
    status: v.union(
      v.literal("new"),
      v.literal("in_progress"),
      v.literal("contractor_scheduled"),
      v.literal("awaiting_follow_up"),
      v.literal("resolved")
    ),
    contractorForTheJob: v.optional(v.string()),
    resolvedAt: v.optional(v.number()), // unix ms — set when status moves to "resolved"
  })
    .index("by_agency", ["agencyId"])
    .index("by_agency_status", ["agencyId", "status"])
    .index("by_property", ["propertyId"])
    .index("by_contact", ["contactId"])
    .index("by_call", ["callId"]),

  media: defineTable({
    agencyId: v.id("agencies"),
    issueId: v.id("issues"),
    type: v.union(v.literal("photo"), v.literal("video")),
    storageId: v.id("_storage"),
    uploadedByUserId: v.string(), // Clerk user ID
    scheduledForDeletionAt: v.optional(v.number()), // unix ms — set by retention job
  })
    .index("by_issue", ["issueId"])
    .index("by_agency", ["agencyId"])
    .index("by_scheduledForDeletion", ["scheduledForDeletionAt"]),
});
```

**Notes:**
- `agencyId` on every tenant-scoped table. Every query filters by it. See section 5 for how it's derived from the Clerk user.
- Clerk user IDs are strings (e.g. `user_xxx`), stored as `v.string()`.
- `Id<"_storage">` is Convex's built-in type for its file storage.
- No Users table — Clerk is the source of truth for users. Agency membership is stored in Clerk's `publicMetadata`.

## 5. Auth and Multi-Tenancy

**Tenancy model: custom, via Clerk user `publicMetadata`.**

Each Clerk user has `publicMetadata.agencyId` set to the Convex `Id<"agencies">` of the Agency they belong to.

### 5.1 Why not Clerk Organizations

Clerk Organizations is a strong fit for multi-user-per-tenant apps but gates behind paid Clerk tiers. Phase 0 has one user per Agency. Custom tenancy via `publicMetadata.agencyId` is simpler and free. Migration path to Organizations is clean if needed in Phase 1 when Team Members are introduced.

### 5.2 How it works

- During onboarding, the operator creates the Agency in Convex, then creates a Clerk user via Clerk's dashboard or API with `publicMetadata: { agencyId: "<convex agency id>" }`.
- On every Convex query/mutation, the function reads the calling user's identity (`ctx.auth.getUserIdentity()`), extracts the `agencyId` from their `publicMetadata`, and scopes queries to that Agency.
- A shared helper `getAgencyIdForCurrentUser(ctx)` lives in `convex/lib/auth.ts`. Every query/mutation that returns tenant data calls this first and throws if it fails.

### 5.3 Auth flow

- Unauthenticated users hit `/sign-in`
- Authenticated users without an `agencyId` in their metadata see a "Not provisioned" page (this state should not exist in practice; if it does, the operator needs to set the metadata)
- Authenticated users with an `agencyId` can access `/open-issues` and `/call-logs`
- Sign up is disabled in Phase 0 (invite-only via operator creating users in Clerk)

### 5.4 Convex + Clerk integration

Standard Convex + Clerk integration per the Convex docs. `convex/auth.config.ts` configured with the Clerk JWT issuer.

## 6. API Surface

Convex functions only. No Next.js API routes in Phase 0 (webhooks go via Convex HTTP actions).

### 6.1 Queries (read)

- `issues.listOpen` — returns Issues where status ≠ `resolved`, grouped by status in the response shape. Scoped to caller's Agency.
- `issues.listResolved` — returns Issues where status = `resolved`. Scoped to caller's Agency.
- `issues.getById` — returns a single Issue with populated Property, Contact, Call (including transcript and audio URL), and Media.
- `issues.search` — full-text-ish search against property address and issue description. Case-insensitive. Scoped to caller's Agency.
- `calls.list` — returns Calls in a date range, ordered by `startedAt` desc. Scoped to caller's Agency.
- `calls.metrics` — returns `{ calls, issuesRecorded, openIssues, passRate }` for a date range. `openIssues` is independent of date range; others respect it.

### 6.2 Mutations (write)

- `issues.updateStatus` — updates an Issue's status. If new status is `resolved`, sets `resolvedAt`. Scoped.
- `issues.updateContractor` — updates `contractorForTheJob`. Scoped.
- `media.createUploadUrl` — returns a Convex storage upload URL for client-side upload. Scoped (caller must own the target Issue's Agency).
- `media.attachToIssue` — after client uploads to the URL, this creates the `media` row linking the storage ID to an Issue. Scoped.

### 6.3 HTTP actions

- `POST /elevenlabs/webhook` — see section 7.

### 6.4 Scheduled jobs (crons)

- `crons.markMediaForDeletion` — daily. Finds media on Issues resolved more than 90 days ago with `scheduledForDeletionAt` unset, sets it to `now + 7 days`.
- `crons.hardDeleteMedia` — daily. Finds media with `scheduledForDeletionAt` in the past, deletes from Convex storage, removes the row.

## 7. ElevenLabs Integration

The Property Management AI Agent is managed in ElevenLabs. This app only receives call results via a post-call webhook.

### 7.1 Webhook endpoint

- URL: `https://<convex-prod-url>/elevenlabs/webhook`
- Method: POST
- Auth: shared secret in header `X-ElevenLabs-Secret`, validated against `ELEVENLABS_WEBHOOK_SECRET` env var in Convex
- Content-Type: `application/json`

### 7.2 Expected payload (target shape)

```ts
{
  callId: string;                    // ElevenLabs call identifier
  agentId: string;                   // the ElevenLabs agent that handled the call
  startedAt: string;                 // ISO 8601
  durationSeconds: number;
  audioUrl: string;                  // ElevenLabs-hosted recording URL
  transcript: Array<{
    speaker: "agent" | "caller";
    text: string;
  }>;
  capturedDetails: {                 // populated by the agent's structured output
    address?: string;
    tenantName?: string;
    tenantContactNumber?: string;
    description?: string;
    urgent?: boolean;
  };
}
```

**The actual ElevenLabs payload format may differ.** The ElevenLabs integration ticket includes confirming the real shape and writing an adapter between their payload and this internal shape.

### 7.3 Webhook handler behaviour

1. Validate `X-ElevenLabs-Secret` header. Reject 401 if mismatch.
2. Look up the Agency by `agentId` (the `elevenLabsAgentId` field on `agencies`).
3. Create a `calls` row with the transcript, audio URL, duration, and `startedAt`.
4. Evaluate `callStatus`:
   - All four `capturedDetails` fields present and non-empty → `success`
   - Otherwise → `failed`
5. If `success`:
   - Match the property: find a Property in this Agency whose address loosely matches `capturedDetails.address`. Matching logic is best-effort — if no match, leave `propertyId` unset and flag the Issue as "unmatched" (needs a flag on the Issue schema — see section 7.5). For Phase 0, assume the operator has imported the Property correctly and the AI Agent captures the address accurately enough to match.
   - Match the contact: find a Contact in this Agency whose phone matches `capturedDetails.tenantContactNumber`. Same best-effort behaviour.
   - Create an `issues` row with status `new`, the captured Details, `urgent` from the payload.
   - Set `calls.issueId` to the new Issue's ID.
6. Return 200 OK.

### 7.4 Building without the webhook wired up

**All app code can be built and tested before ElevenLabs is connected.** The first Convex seed script (`convex/seed.ts`) creates Agency, Property, Contact, Call, and Issue records using the same shape the webhook would produce. Coding agents use this to develop and test.

The ElevenLabs integration ticket is late in the sequence — after the UI, data layer, and mock data all work. It only needs to wire the real webhook to an internal function that already exists and has been tested.

### 7.5 Unmatched-property handling in Phase 0

If the AI Agent captures an address that doesn't match any Property in the Agency, the current shape would fail. For Phase 0, **assume this doesn't happen** — the AI Agent's prompt enforces address verification during the call. If it does happen:
- Log the failure in Convex logs
- Create the Call with `callStatus = "failed"` and no Issue
- Operator can review Convex logs and intervene manually

No UI for unmatched Issues in Phase 0.

## 8. Client-side patterns

### 8.1 Data fetching

- Use Convex React hooks (`useQuery`, `useMutation`) for all data access.
- No separate API layer — components read directly from Convex queries.
- Convex handles real-time updates: when an Issue's status changes, the Open Issues list reactively updates without a refetch.

### 8.2 Routing

- `/sign-in` — Clerk sign-in
- `/` (root) — redirects to `/open-issues` if signed in, to `/sign-in` if not
- `/open-issues` — Open Issues list
- `/open-issues/[issueId]` — Open Issues list with the slide-over panel open to the specified Issue (the URL represents panel state)
- `/call-logs` — Call Logs page

The `/open-issues/[issueId]` pattern means an Issue URL can be shared and opens the panel over the list.

### 8.3 State management

- Server state: Convex (real-time, no cache management needed).
- UI state (open group sections, current date range selection, search input value): React `useState` or URL search params where the state should survive a reload (e.g. date range on Call Logs).

### 8.4 Component organisation

Atomic pattern as defined in folder structure (section 2.2):
- `/components/ui` — primitives, no business logic
- `/components/composed` — combinations, still generic
- `/components/features` — Phase 0 features (OpenIssuesList, IssuePanel, CallLogsTable, etc.)

Reuse before building. If a primitive exists in `ui`, use it.

## 9. Media handling

### 9.1 Upload flow

1. Admin clicks Upload Media in the Issue panel.
2. Client calls `media.createUploadUrl` (mutation) to get a one-time upload URL from Convex storage.
3. Client POSTs the file directly to that URL.
4. Client calls `media.attachToIssue` with the returned storage ID to create the `media` row.
5. UI refreshes — the new media appears in the Issue panel.

### 9.2 Display

- Photos: inline thumbnails, click to open full-size in a modal or new tab.
- Videos: thumbnail with a play icon, click to open a video player.
- All media for an Issue is fetched with the Issue detail query.

### 9.3 Retention

Two scheduled jobs in `convex/crons.ts` (see section 6.4). Deletion is permanent — no undo — but has a 7-day buffer after marking, giving a recovery window.

### 9.4 Audio recordings

Phase 0 stores the ElevenLabs-hosted URL on the `calls.audioUrl` field and plays it directly in the Issue panel. No download into Convex storage unless the URL proves unreliable — if so, the integration ticket should download and cache it.

## 10. Testing

### 10.1 What gets tested

**Unit tests (required for all new Convex functions):**
- Queries: correct data returned, correct scoping by Agency, correct filtering.
- Mutations: correct state change, correct authorization.
- Webhook action: valid payload creates Issue; invalid payload returns 400; missing secret returns 401; failed calls (missing fields) create Call without Issue.

**Integration tests (one per critical flow):**
- ElevenLabs webhook → Call created → Issue created → appears in `issues.listOpen`
- Media upload → attached to Issue → appears in Issue detail query
- Media retention job → marks → deletes

**Manual verification (no automated tests for Phase 0):**
- UI flows (Open Issues rendering, panel interactions, browsing with up/down)
- Clerk sign-in flow
- Cross-Agency data isolation (spot-check with two seeded Agencies)

### 10.2 Testing framework

- Convex: `convex-test` for Convex function testing
- React: Vitest for any non-Convex logic
- No Playwright / E2E in Phase 0

### 10.3 TDD expectation

For Convex functions: write the test first, then the function. Each PR includes tests for the code it introduces.

For UI: coding agents are trusted to implement without tests in Phase 0. Manual verification covers it.

## 12. Conventions for coding agents

- **TypeScript strict mode on.** All new code is fully typed. No `any` without justification.
- **File size.** A component file should be under ~200 lines. If it grows past that, split.
- **Import order.** External libs → Convex → app imports (`@/...`) → styles. Prettier handles the rest.
- **Naming.**
  - Convex tables: plural camelCase (`issues`, `calls`)
  - Convex functions: `moduleName.functionName` (`issues.listOpen`)
  - React components: PascalCase, one component per file
  - Utilities: camelCase
- **Comments.** Code should explain what, comments explain why. No comments that restate the code.
- **Error handling.** Convex functions throw on error; clients use `useQuery` / `useMutation` error states.
- **No `console.log` in committed code.** Use Convex logs (`console.log` inside a Convex function is fine) or remove before commit.

## 15. Open items for the coding agents to resolve

Things that need answering during implementation, not upfront:

- **Exact ElevenLabs webhook payload format.** Spec defines the target shape; the real payload may differ and needs adapting. First task of the ElevenLabs integration ticket.
- **Address matching logic.** `capturedDetails.address` might be "59 Wakefield Road" and the stored Property might be "59 Wakefield Road, Hipperholme, Halifax, HX3 8AQ". Fuzzy match on postcode + street? On exact string? Implement simplest viable matching; log unmatched for operator review.
- **Phone number normalisation.** UK numbers come in multiple formats (`07729420529`, `+447729420529`, `07729 420 529`). Normalise to one canonical form before matching. Use a lib like `libphonenumber-js` if simple normalisation isn't enough.
- **Audio player UI.** Native `<audio>` element or a styled player? Defer to implementation.
- **Video player UI.** Same.
- **Search debouncing on Open Issues.** Not needed at MVP scale but worth a note — if the list grows past ~200 Issues, revisit.
